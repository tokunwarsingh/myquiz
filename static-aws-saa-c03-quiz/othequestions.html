🌐 AWS Global Infrastructure

Regions, AZs & Edge Locations

  • Region: physical location with 3+ isolated AZs — e.g. us-east-1
  • AZ: one or more discrete data centers with redundant power/networking. Connected via low-latency private links
  • Edge Locations / PoPs: 400+ worldwide — used by CloudFront, Route 53, Global Accelerator
  • Local Zones: AWS infra closer to large population centres (single-digit ms latency)
  • Wavelength Zones: embed AWS compute inside telecom 5G networks
  • Outposts: AWS rack delivered to your data centre — fully managed by AWS

Global Services (not region-specific)

  • IAM, CloudFront, Route 53, WAF, Shield, Organizations, Billing
  • Everything else is region-scoped

⚠️ Exam Traps

  • Edge Locations ≠ AZs. ELs are for caching/routing; AZs are for HA compute
  • S3 bucket names global but data stored in the region you create it
  • When choosing region: compliance → latency → service availability → pricing

🔒 IAM & Security

Core IAM Objects

  • Users: individuals. Max 5,000/account. Have long-term credentials (password + access keys)
  • Groups: collection of users. No nesting. Cannot be principal in resource policy
  • Roles: temp credentials via STS. No password/access keys stored. Used by EC2, Lambda, cross-account
  • Policies: JSON permission docs. Explicit DENY always wins. Default = implicit DENY
  • Instance Profile: wrapper around IAM Role so EC2 can assume it. Never hardcode credentials on EC2

Policy Types

  • Identity-based: attached to user/group/role
  • Resource-based: attached to resource (S3 bucket policy, SQS, SNS). Requires Principal field
  • Permission Boundary: sets MAXIMUM permissions — does NOT grant anything by itself
  • SCP (Organizations): max permissions for entire account/OU. Management account is NOT affected by SCPs
  • Session Policies: passed when assuming a role via CLI/API to further restrict

Policy Evaluation Order

  1. Explicit DENY  → DENY (always wins, anywhere in chain)
  2. SCP            → must ALLOW (otherwise DENY)
  3. Resource policy → check (cross-account: both must allow)
  4. Permission Boundary → must ALLOW
  5. Identity policy → must ALLOW
  ✓ → ACCESS GRANTED

🗺️ IAM Policy Evaluation Flowchart

STS & Temporary Credentials

  • AssumeRole: default 1 hr, max 12 hrs. Returns: AccessKeyId + SecretAccessKey + SessionToken
  • AssumeRoleWithWebIdentity: OIDC federation (Google, Facebook, Cognito)
  • AssumeRoleWithSAML: enterprise SSO via SAML 2.0
  • GetSessionToken: MFA-protected operations for existing user
  • Cross-account: Target creates role with trust policy for source account → source calls AssumeRole
  • Resource-based policy (S3) does NOT require giving up own permissions, unlike AssumeRole

Common IAM Conditions

  • aws:SourceIp — restrict by client IP
  • aws:RequestedRegion — restrict which region can be called
  • aws:MultiFactorAuthPresent — require MFA
  • aws:PrincipalOrgID — restrict to AWS Organization members
  • aws:ResourceTag / aws:PrincipalTag — tag-based ABAC
  • ec2:ResourceTag — restrict EC2 actions to tagged resources

IAM Access Analyzer

  • Identifies resources shared with external entities (outside org/account)
  • Analyzes: S3, IAM roles, KMS keys, Lambda, SQS, Secrets Manager
  • Policy Validation: check policies before applying
  • Policy Generation: watch CloudTrail activity → generate least-privilege policy

AWS Organizations & SCPs

  • Management account + Member accounts grouped into OUs
  • Consolidated billing: shared Reserved Instances & Savings Plans discounts
  • SCPs: DENY-only at OU/account level. Management account always exempt
  • SCPs don't grant permissions — they only restrict what IAM can grant
  • Use aws:PrincipalOrgID condition in resource policies to allow all org members

IAM Identity Center (SSO)

  • Single sign-on across multiple AWS accounts & SaaS apps
  • Permission Sets = collection of policies assigned to users/groups per account
  • Supports: AWS Identity Center directory, Active Directory, external IdP (Okta, Azure AD)
  • Requires AWS Organizations

⚠️ Exam Traps

  • Root user: never use daily, never create access keys, enable MFA immediately
  • Groups CANNOT be principals in resource-based policies
  • Permission Boundary + Identity policy: effective = intersection (both must allow)
  • Cross-account: BOTH identity policy in source AND resource policy (or role) in target must allow
  • SCP on management account = no effect

🗺️ EC2 Placement Group Types

Security Groups

  • Stateful firewall at ENI level. Only ALLOW rules (no explicit deny)
  • Default: all inbound denied, all outbound allowed
  • Can reference another SG by ID (not name) — allows traffic from instances in that SG
  • Max 5 SGs per ENI, 60 rules per direction

EC2 Hibernate

  • Saves RAM state to encrypted root EBS → stops instance. Resumes faster than cold reboot
  • Requires: EBS-backed + encrypted root volume + instance type support
  • Use case: long initialization processes (e.g. pre-warm ML model)
  • Processes are still "running" from OS perspective on resume

ENI (Elastic Network Interface)

  • Virtual NIC attached to instance. Has private IP, optional public/Elastic IP, MAC address, SGs
  • Can attach multiple ENIs → dual-homed instance (in different subnets)
  • ENI is AZ-scoped; can be moved between instances in same AZ for failover
  • Elastic IP: static public IP — costs $0.005/hr when NOT associated with running instance

User Data & Metadata

  • User Data: bootstrap script run ONCE at first launch (root privileges). Max 16 KB
  • Metadata: http://169.254.169.254/latest/meta-data/ — get instance ID, IAM role credentials, etc.
  • IMDSv2 (preferred): requires session token, protects against SSRF

⚠️ Exam Traps

  • Spot: can be terminated with only 2-min notice — NOT for critical/long-running services
  • Capacity Reservations guarantee capacity but you pay whether you use it or not
  • Cluster PG: single-AZ only — entire cluster fails if AZ fails
  • Spread PG: max 7 per AZ — use Partition if you need more
  • Dedicated Host ≠ Dedicated Instance. Host = physical server visibility; Instance = isolated hardware

EBS Key Facts

  • Bound to a single AZ. Move between AZs via snapshot → create volume in new AZ
  • Can resize volume without stopping instance (for gp2/gp3/io1/io2)
  • Encryption: AES-256 at rest & in transit. Cannot disable after creation
  • To encrypt unencrypted volume: snapshot → copy snapshot with encryption enabled → restore
  • Multi-Attach: io1/io2 only, max 16 instances in same AZ — requires cluster-aware FS
  • Default: Root EBS volume deleted on termination; extra volumes persist

EBS Snapshots

  • Point-in-time backup stored in S3 (not visible in S3 console). Incremental — only changed blocks
  • Can copy across regions for DR. Can share with other accounts
  • Fast Snapshot Restore (FSR): eliminates latency on first use — costs extra
  • Recycle Bin: protects against accidental deletion (1 day – 1 year retention)
  • EBS Snapshot Archive: 75% cheaper, but 24–72 hr restore time

EFS (Elastic File System)

  • Managed NFS (NFSv4) — mounts on multiple EC2 instances across AZs simultaneously
  • Linux only. Pay per GB used. Scales to petabytes automatically
  • Performance modes: General Purpose (default, low latency) vs Max I/O (highly parallel)
  • Throughput modes: Bursting (scales with size) vs Provisioned (specify MB/s)
  • Storage tiers: Standard vs Infrequent Access (EFS-IA, up to 92% cheaper)
  • EFS One Zone: single AZ, 47% cheaper than Standard — good for dev/test

Instance Store

  • Physically attached NVMe SSD — highest IOPS possible (millions), zero latency
  • Ephemeral: data LOST on stop/terminate/hardware failure
  • Cannot be detached or resized. Included in instance price
  • Use for: buffer/cache/scratch space/temp data only

⚠️ Exam Traps

  • gp2 IOPS = 3× GB (small volume = low IOPS). gp3 always 3,000 IOPS regardless of size
  • st1/sc1 (HDD) cannot be boot volumes
  • EFS is NOT for Windows (NFS only). Use FSx for Windows instead
  • Instance Store data lost on stop — but NOT on reboot
  • Snapshot import to Glacier: must go S3 first → lifecycle policy to Glacier

🗺️ Load Balancer Types by OSI Layer

ALB Details

  • Target Groups: EC2 instances, ECS tasks, Lambda functions, IP addresses
  • Rules: path-based (/api/*), host-based (api.example.com), header, query string, source IP
  • Stickiness: load balancer-generated cookie (AWSALB) — binds user to same target
  • Connection Draining (Deregistration Delay): default 300s (0-3600s). Waits for in-flight requests before deregistering
  • X-Forwarded-For header: original client IP. X-Forwarded-Port, X-Forwarded-Proto
  • SSL/TLS: SNI (Server Name Indication) — multiple TLS certs on one LB for multiple domains

NLB Details

  • Static IP per AZ (or Elastic IP). Source IP preserved (client IP visible to target)
  • Ultra-low latency. No security group on NLB itself (inbound rules on targets must allow 0.0.0.0/0)
  • Can target ALB for static IP + L7 routing combo

Cross-Zone Load Balancing

LBDefaultCost
ALBEnabledNo inter-AZ charge
NLB / GWLBDisabledCharged if enabled
CLBDisabledNo charge if enabled

Health Checks

  • Healthy threshold (default 5 checks), unhealthy threshold (default 2)
  • Interval: 5–300 seconds. If 2xx/3xx → healthy. Configurable per target group
  • ALB: HTTP/HTTPS. NLB: TCP/HTTP/HTTPS. GWLB: HTTP/HTTPS/TCP

ASG Scaling Policies

PolicyTriggerBest For
Simple ScalingCloudWatch alarm → add/remove NBasic scaling

Target TrackingKeep metric at target (e.g. CPU=50%)Most common, self-adjusting

ASG Key Concepts

  • Cooldown Period: default 300s after scale activity — prevents rapid oscillation
  • Lifecycle Hooks: pause instance in Pending (launch hook) or Terminating state to run custom scripts
  • Warm Pools: pre-initialized stopped instances for faster scale-out
  • Launch Template (preferred over Launch Config): versioned, supports Spot + On-Demand mix
  • ASG free — you pay for the underlying EC2 instances
  • Min size / Desired capacity / Max size
  • Scale-in protection: prevent specific instances from being terminated during scale-in

⚠️ Exam Traps

  • NLB preserves source IP — ALB replaces with LB IP (use X-Forwarded-For)
  • Only NLB can have Elastic IP assigned per AZ
  • Cross-zone LB on NLB has inter-AZ data transfer cost (not on ALB)
  • ALB cannot have static IP — put NLB in front of ALB to get static IPs
  • Connection draining ≠ health check. Draining allows in-flight requests to complete
  • Target Tracking will not scale in if metric is above target AND instances are unhealthy

🗺️ Multi-AZ vs Read Replica Architecture

RDS Key Facts

  • Managed: automated backups (1-35 days), Multi-AZ, patching, snapshots
  • Engines: MySQL, PostgreSQL, MariaDB, Oracle, SQL Server, IBM Db2
  • RDS Custom: OS & DB customization for Oracle & SQL Server
  • Storage autoscaling: auto-increases when <10% free, max threshold configurable
  • Encryption: at-rest with KMS. Must enable at creation. Encrypted snapshot → encrypted restore only
  • IAM Authentication: supported for MySQL & PostgreSQL (token-based, 15-min lifetime)
  • RDS Proxy: pools connections, reduces DB load. Required for Lambda → RDS. Enforces IAM auth. Only accessible within VPC

Aurora

  • AWS-proprietary. PostgreSQL & MySQL compatible. 5× MySQL, 3× PostgreSQL performance
  • Storage auto-grows 10GB → 128TB in increments. 6 copies across 3 AZs (4/6 writes, 3/6 reads)
  • Up to 15 read replicas (vs 5 for RDS). Replica failover <30s
  • Aurora Serverless: auto-scales ACUs (Aurora Capacity Units). Good for infrequent/unpredictable workloads
  • Aurora Global: 1 primary region + up to 5 secondary regions. <1s replication lag. RPO <1s, RTO <1min
  • Aurora Multi-Master: all nodes can write — continuous write availability
  • Backtrack: rewind DB to any point in the past without restoring from backup
  • Custom endpoints: route different read replica sets to different applications

Caching Strategies

  • Lazy Loading (Cache-Aside): check cache → miss → load from DB → populate cache. Stale data possible
  • Write-Through: write to cache AND DB simultaneously. No stale data. Cache churn if data rarely read
  • Session Store: store user sessions in ElastiCache with TTL for stateless app scaling

⚠️ Exam Traps

  • Read Replica → can be promoted to standalone DB (breaks replication)
  • Multi-AZ failover: DNS updated to point to standby — app must use DNS endpoint, not IP
  • Aurora replicas can failover automatically; RDS Read Replicas need manual promotion
  • RDS Proxy: ONLY within VPC. Use for Lambda connections to avoid connection storms
  • Encrypted RDS: cannot create unencrypted Read Replica from encrypted source
  • ElastiCache in front of RDS: always check cache first, then DB (read-heavy workloads)

🌐 Route 53

Record Types

  • A: hostname → IPv4. AAAA: hostname → IPv6
  • CNAME: hostname → hostname. Cannot be root domain (apex). Charged per query
  • Alias: hostname → AWS resource. Works at root domain. Free queries. No TTL control
  • NS: name servers for hosted zone
  • Alias targets: ALB, NLB, CloudFront, API GW, Elastic Beanstalk, S3 website, VPC Endpoint. NOT EC2 DNS name

Routing Policies

PolicyHealth CheckBest ForKey Detail
Simple❌Single resourceMultiple values → client picks randomly
Weighted✅A/B testing, gradual migrationWeight 0 = no traffic. All 0 = equal
Latency✅Lowest latency to userBased on AWS region latency, not geography
Failover✅ (mandatory on primary)Active-passive DRSecondary used only if primary unhealthy
Geolocation✅Content localizationBy continent/country/US state. Need Default record
Geoproximity✅Shift traffic by biasRequires Traffic Flow. Bias +1 to +99 / -1 to -99
Multi-Value✅Client-side LBUp to 8 healthy records returned. NOT a substitute for ELB
IP-Based✅ISP/CIDR routingMap CIDR → endpoint. Optimize peering costs

Health Checks

  • ~15 global health checkers. Interval: 30s (or 10s for fast, higher cost)
  • Healthy if >18% checkers report healthy
  • Checks: HTTP/HTTPS/TCP. 2xx or 3xx = healthy. Can match first 5,120 bytes of response body
  • Calculated Health Checks: combine up to 256 child checks with AND/OR/NOT
  • Private resources: health checkers are outside VPC → use CloudWatch Alarm → Route 53 checks alarm state

TTL & Hosted Zones

  • TTL mandatory for all records except Alias. High TTL = less cost but stale data. Low TTL = agile but costly
  • Public Hosted Zone: accessible from internet
  • Private Hosted Zone: within VPC only. Must enable VPC DNS resolution + DNS hostnames settings

⚠️ Exam Traps

  • CNAME cannot point to root domain (mydomain.com). Use Alias
  • Geolocation ≠ Latency. Geo = physical location. Latency = measured network speed
  • Geoproximity requires Route 53 Traffic Flow (extra cost)
  • Simple routing policy cannot have health checks
  • Multi-Value ≠ ELB. It's just returning multiple healthy IPs to the client
  • Alias record type is always A or AAAA (not CNAME)

S3 Core Limits

  • Max object size: 5 TB. Single PUT: max 5 GB
  • Multipart upload recommended >100 MB, required >5 GB
  • Bucket names: globally unique, 3-63 chars, lowercase, no underscores
  • Versioning: once enabled, can only be suspended (not disabled)
  • MFA Delete: requires MFA to delete versioned objects or suspend versioning. Root user only can enable

Replication

  • CRR (Cross-Region): DR, latency reduction, compliance. Must enable versioning on both buckets
  • SRR (Same-Region): log aggregation, test/prod sync
  • Replication is asynchronous. Does NOT replicate existing objects (only new after enabling). Deletes NOT replicated by default
  • Can replicate delete markers (opt-in). Cannot replicate SSE-C encrypted objects

Encryption

Access Control

  • Bucket Policy: JSON resource-based policy. Controls cross-account access, enforce HTTPS, enforce SSE-KMS
  • ACLs: legacy. Disabled by default on new buckets (Block Public Access preferred)
  • S3 Access Points: per-application named access policies. Simplifies bucket policy management
  • Object Lambda Access Points: run Lambda on GET requests to transform data on-the-fly
  • Pre-signed URL: time-limited access to private objects. Generated by any AWS principal

Performance

  • Baseline: 3,500 PUT/COPY/POST/DELETE and 5,500 GET/HEAD per second per prefix
  • Transfer Acceleration: upload via nearest CloudFront edge → AWS backbone. Charged extra
  • S3 Select: SQL to retrieve subset of object (CSV, JSON, Parquet). Up to 400% faster, 80% cheaper vs full download
  • Byte-Range Fetches: parallel downloads of specific byte ranges for large files

Event Notifications

  • Triggers on: s3:ObjectCreated, s3:ObjectRemoved, s3:Replication, etc.
  • Destinations: SNS, SQS, Lambda, EventBridge (most feature-rich)
  • EventBridge: advanced filtering, multiple destinations, archive & replay

S3 Lifecycle Rules

  • Transition actions: move to cheaper class after N days
  • Expiration actions: delete objects after N days, or delete old versions
  • Can apply to specific prefix or tags. Minimum days before transitioning: Standard-IA/One Zone-IA requires 30 days in Standard first

🗺️ S3 Storage Class Transitions

⚠️ Exam Traps

  • SSE-KMS: every object GET/PUT = 1 KMS API call — can hit KMS throttle at high throughput
  • S3 is not a file system — no true directories, no atomic renames
  • Replication does NOT copy existing objects — use S3 Batch Operations for that
  • One Zone-IA: data lost if AZ fails — never for primary storage of important data
  • Glacier Flexible retrieval: Expedited (1-5 min), Standard (3-5 hrs), Bulk (5-12 hrs)
  • Transfer Acceleration: only beneficial over long distances — test before enabling

🌍 CloudFront & Global Accelerator

CloudFront Key Facts

  • CDN with 400+ edge locations. Caches content at edge for TTL duration
  • Origins: S3, ALB, EC2, any HTTP endpoint, MediaStore
  • OAC (Origin Access Control): restricts S3 to CloudFront only (preferred over OAI)
  • Cache Invalidation: force refresh before TTL — use /* (all) or path pattern
  • Geo Restriction: allowlist or blocklist by country (3rd-party Geo-IP database)

🗺️ SNS Fan-out → SQS Pattern

SQS Key Facts

  • Message size: max 256 KB (use S3 + Extended Client Library for larger)
  • Retention: 60s – 14 days (default 4 days)
  • Visibility Timeout: 0–43,200s (default 30s). Message invisible while processing. Too short = reprocessing; too long = slow failure detection
  • Dead Letter Queue (DLQ): receives messages after N failed processing attempts (maxReceiveCount)
  • Long Polling: wait up to 20s for messages (reduces empty responses & cost). Always prefer over short polling
  • Delay Queue: delay delivery up to 15 min
  • Consumer can delete message only after successful processing (using ReceiptHandle)

SNS

  • Pub/Sub. Publisher → Topic → Subscribers (SQS, Lambda, HTTP/S, Email, SMS, Mobile Push)
  • Messages NOT persisted (fire and forget). Max 256 KB
  • Message Filtering: JSON filter policy per subscription — only matching messages delivered
  • Fan-Out Pattern: SNS → multiple SQS queues (decouple producers from consumers)
  • FIFO SNS: ordered + deduplicated. Only FIFO SQS as subscriber. Same throughput limits
  • DLQ available for Lambda/SQS subscribers

Kinesis Data Streams Details

  • Shard: 1 MB/s or 1,000 records/s write; 2 MB/s read per shard
  • Record: partition key + sequence number + data blob (max 1 MB)
  • Enhanced Fan-Out: each consumer gets dedicated 2 MB/s per shard (vs shared 2 MB/s for classic)
  • Retention: default 24 hrs, extendable to 365 days
  • On-Demand mode: auto-scales, pays per GB. Provisioned mode: pay per shard-hour

Amazon MQ

  • Managed Apache ActiveMQ or RabbitMQ. Protocols: AMQP, MQTT, STOMP, OpenWire, WebSocket
  • Use when: migrating existing on-prem message broker to cloud (NOT for new greenfield apps)
  • Active/Standby for HA. NOT serverless — runs on EC2 instances
  • NOT as scalable as SQS/SNS. Use SQS/SNS for new AWS-native apps

⚠️ Exam Traps

  • SQS: consumer must explicitly delete message. Visibility timeout ≠ message deletion
  • SNS does NOT store messages — if subscriber is down, message is lost (unless SQS in between)
  • Kinesis Firehose is NOT real-time (60s buffer minimum)
  • FIFO SQS: MessageGroupId required; without it, all messages in one group = no parallelism
  • Kinesis shard hot partition: bad partition key design → throttling. Use random suffix or UUID

ECS Key Concepts

  • Task Definition: blueprint — image, CPU/memory, environment vars, ports, volumes, IAM role
  • Task Role: IAM permissions for the running container (e.g. access S3)
  • Execution Role: IAM permissions for ECS agent to pull images from ECR, write logs to CloudWatch

  • Fargate CPU: 256–30,720 CPU units. Memory: 512 MB–120 GB
  • ECS Anywhere: run ECS tasks on on-prem servers

EKS (Elastic Kubernetes Service)

  • Managed Kubernetes control plane. Worker nodes: EC2 managed node groups or Fargate pods
  • IRSA (IAM Roles for Service Accounts): fine-grained IAM per pod — preferred over node IAM role
  • Add-ons: VPC CNI (assigns VPC IPs to pods), CoreDNS, kube-proxy, EBS/EFS CSI driver
  • Use EKS when: already using Kubernetes, multi-cloud portability needed
  • Use ECS when: AWS-native, simpler setup, lower operational overhead

ECR (Elastic Container Registry)

  • Private Docker registry. Integrated with ECS/EKS. Images encrypted at rest (KMS)
  • Lifecycle policies: auto-expire old image tags. Scan on push for vulnerabilities

⚠️ Exam Traps

  • Task Role ≠ Execution Role. Task = app permissions. Execution = ECS agent/infrastructure
  • Fargate: cannot use EC2 placement groups or dedicated hosts
  • ECS on EC2: instances must have ECS agent running and be in ECS cluster
  • ALB dynamic port mapping: ECS registers random host port — ALB handles routing (no static port needed)

Lambda Best Practices

  • Initialize SDK clients and DB connections outside the handler (reused across warm invocations)
  • Use /tmp for temp storage within execution (persists across warm invocations)
  • Set environment variables for configuration. Use Secrets Manager for secrets

  • Event Source Mapping (SQS): Lambda polls queue. Not push — Lambda fetches messages
  • Reserved concurrency = 0 → function is effectively disabled (all invocations throttled)

📦 DynamoDB

Core Concepts

  • Serverless, fully managed NoSQL. Single-digit millisecond performance at any scale
  • Partition Key: required, determines data distribution. Must be unique (if no sort key)
  • Sort Key: optional, enables range queries. PK + SK together must be unique
  • Max item size: 400 KB
  • Supports: strings, numbers, binary, sets, lists, maps, booleans, null

Capacity Modes

ProvisionedOn-Demand
CostCheaper for predictable loadMore expensive per request
ScalingManual or auto-scalingInstant auto-scale
Best forKnown, steady trafficUnpredictable or spiky traffic

RCU / WCU Formulas

  • 1 RCU = 1 strongly consistent read/s of item ≤4 KB OR 2 eventually consistent reads/s
  • 1 WCU = 1 write/s of item ≤1 KB
  • Example: 10 KB item read (strongly consistent) = 3 RCU (ceil(10/4))
  • Example: 10 KB item write = 10 WCU

Indexes

LSI (Local Secondary Index)GSI (Global Secondary Index)
When createdAt table creation ONLYAnytime
Partition keySame as tableAny attribute
CapacityShares table RCU/WCUSeparate RCU/WCU (extra cost)
Max size per PK10 GBUnlimited
Strongly consistent reads?✅ Yes❌ Eventually consistent only

Advanced Features

  • DAX (DynamoDB Accelerator): in-memory cache, microsecond reads. Write-through. API-compatible. No code change needed
  • DynamoDB Streams: ordered stream of item changes (24-hr retention). Trigger Lambda for event-driven patterns
  • Global Tables: multi-region active-active replication. Uses DynamoDB Streams. Requires On-Demand or auto-scaled provisioned
  • TTL: automatically delete items after a timestamp attribute expires. No RCU/WCU consumed
  • Transactions: ACID across multiple items/tables. Max 25 items or 4 MB per transaction. Costs 2×RCU/WCU
  • Point-in-Time Recovery (PITR): continuous backups, restore to any second in last 35 days

⚠️ Exam Traps

  • Scan = reads entire table (expensive). Always use Query with partition key
  • LSI: 10 GB limit per partition key value — plan data modelling carefully
  • GSI eventually consistent only — cannot guarantee immediate read-your-writes
  • Hot partitions: poor partition key (e.g. status field with only 3 values) → throttling
  • TTL deletion is async (up to 48 hr delay). Filter TTL-expired items in application
  • Global Tables: eventual consistency. Each region can write — no automatic conflict resolution

📊 Monitoring (CloudWatch / CloudTrail / Config / X-Ray)

CloudWatch Overview

  • Metrics: time-series data. Default EC2 metrics every 5 min. Detailed monitoring: 1 min (extra cost)
  • Custom Metrics: push via PutMetricData API. High-resolution: 1s granularity
  • Retention: 3 hrs (1s), 15 days (1 min), 63 days (5 min), 15 months (1 hr)
  • Dimensions: up to 30 per metric. Key-value pairs to filter (e.g. InstanceId)

CloudWatch Alarms

  • States: OK, ALARM, INSUFFICIENT_DATA
  • Actions: SNS notification, Auto Scaling policy, EC2 action (stop/terminate/reboot/recover)
  • Composite Alarms: AND/OR logic across multiple alarms — reduce alarm noise
  • Anomaly Detection: ML baseline → alarm when metric deviates from expected band
  • EC2 Instance Recovery: same instance ID, same private IP, same Elastic IP, same metadata

CloudWatch Logs

  • Log Groups → Log Streams → Log Events
  • Retention: 1 day – never expire (default = never expire = costs $$$)
  • Metric Filters: extract metrics from log patterns (e.g. count ERROR occurrences)
  • Logs Insights: interactive SQL-like query engine. Query across multiple log groups
  • Subscription Filters: stream logs to Kinesis, Lambda, Firehose in near real-time
  • Live Tail: real-time log streaming in console
  • Send to: S3 (export, batch), Kinesis Firehose (near real-time to S3/Redshift/OpenSearch)

CloudWatch Agent

  • Required for: memory utilization, disk space, custom app metrics on EC2
  • Unified Agent: sends both metrics AND logs. Configure via SSM Parameter Store
  • Without agent: EC2 only sends CPU, Network, Disk I/O (NOT memory, NOT disk utilization)

CloudTrail

  • API audit log — WHO called WHAT API from WHERE at WHEN
  • Console in CloudTrail: 90-day retention. Long-term: send to S3
  • Management Events: default enabled. API calls that configure resources (CreateBucket, RunInstances)
  • Data Events: off by default. High-volume ops: S3 object-level (GetObject, PutObject), Lambda invocations
  • CloudTrail Insights: detects unusual write activity using ML
  • Log file validation: SHA-256 hash — detect tampering
  • Organization Trail: one trail for all accounts in an AWS Organization
  • Multi-region trail: single trail covering all regions

AWS Config

  • Records configuration state of AWS resources over time. Answers: "Was my SG open to 0.0.0.0/0 last Tuesday?"
  • Config Rules: evaluate compliance. AWS managed (75+) or custom (Lambda-based)
  • Remediation: auto-fix via SSM Automation documents
  • Per-region service. Use Config Aggregator for cross-account/region view
  • NOT preventive — detects non-compliance after the fact (use SCPs/IAM for prevention)

X-Ray / Other Monitoring

  • X-Ray: distributed tracing. Service map, latency analysis, error root cause. Requires X-Ray SDK + Daemon
  • Sampling: controls what % of requests are traced (reduce cost)
  • EventBridge: event bus. Schedule (cron/rate) or event pattern → target (Lambda, SQS, Step Functions, etc.)
  • EventBridge Schema Registry: discover & version event schemas
  • Trusted Advisor: checks for cost, security, fault tolerance, performance, service limits. Full access needs Business/Enterprise Support
  • Personal Health Dashboard: account-specific service events & planned maintenance

⚠️ Exam Traps

  • EC2 default metrics do NOT include memory or disk — must install CloudWatch Agent
  • CloudTrail Data Events cost extra — not enabled by default
  • Config Rules are detective, NOT preventive. SCPs and IAM deny are preventive
  • CloudWatch Logs default retention = never expire = ongoing storage cost. Always set retention policy
  • Alarm can't go from OK directly to INSUFFICIENT_DATA by alarm logic — only if data stops flowing

🔗 VPC & Networking

VPC Fundamentals

  • Default VPC: 172.31.0.0/16, exists in all regions, all instances get public IPs
  • Max 5 VPCs/region, max 5 CIDRs/VPC (min /28, max /16)
  • AWS reserves 5 IPs per subnet: .0 (network), .1 (router), .2 (DNS), .3 (future), .255 (broadcast)
  • Subnet in single AZ. Public subnet = route to IGW. Private subnet = no direct IGW route

🗺️ Transit Gateway — Hub & Spoke

Direct Connect Details

  • Dedicated Connections: 1/10/100 Gbps. Physical port. Lead time >1 month
  • Hosted Connections: 50 Mbps–10 Gbps via DX Partner. Flexible add/remove
  • Private VIF: accesses VPC via VGW
  • Public VIF: accesses AWS public services (S3, EC2 public IPs)
  • Transit VIF: connects to Transit Gateway
  • DX Gateway: one DX connection → VGWs in multiple regions (same account)
  • Encryption: NOT encrypted by default. Add VPN on top for IPSec encryption
  • Fallback: Use Site-to-Site VPN as backup when DX fails

Transit Gateway & ECMP

  • Transitive routing hub — VPCs can communicate through TGW without direct peering
  • ECMP: route across multiple VPN tunnels for higher aggregate bandwidth
  • 1 VPN = 1.25 Gbps. 2 VPNs with ECMP = 2.5 Gbps. Add more for more bandwidth
  • Cross-region TGW peering supported. Share via Resource Access Manager (RAM)

VPC Flow Logs

  • Captures: src/dst IP, src/dst port, protocol, packets, bytes, action (ACCEPT/REJECT)
  • Enable at VPC, subnet, or ENI level. Does NOT capture DNS, Windows license activation, instance metadata, DHCP
  • Destination: CloudWatch Logs, S3, Kinesis Firehose
  • Query with Athena (S3) or CloudWatch Logs Insights
  • REJECT = SG or NACL blocked. Use to debug connectivity issues

IPv6 & Egress-Only IGW

  • All IPv6 addresses are public (no private IPv6). AWS assigns /56 CIDR to VPC
  • Egress-Only IGW: allows IPv6 outbound from private subnet, blocks inbound — equivalent of NAT for IPv6
  • Dual-stack: instance can have both IPv4 (private/public) and IPv6

⚠️ Exam Traps

  • VPC Peering is NOT transitive. A↔B, B↔C does NOT allow A↔C. Use Transit Gateway instead
  • NAT Gateway: only one AZ. For HA, create one NAT GW per AZ with separate route tables
  • NACLs are stateless: must allow ephemeral ports (1024-65535) for return traffic
  • Gateway Endpoint is FREE — always choose it for S3/DynamoDB access from VPC
  • DX does NOT encrypt — add VPN for encryption over DX
  • VGW required for both Site-to-Site VPN AND Direct Connect to VPC
  • S2S VPN: must enable Route Propagation on route table to use VGW routes

🔐 Security Services

KMS (Key Management Service)

  • AWS Owned Keys: free, AWS manages, no visibility
  • AWS Managed Keys: free, auto-rotates yearly, visible in KMS console
  • Customer Managed Keys (CMK): $1/month + API calls. Full control, custom rotation
  • Symmetric: AES-256. Asymmetric: RSA, ECC (public key downloadable, private stays in KMS)
  • Envelope Encryption: KMS generates Data Key → encrypts data locally → stores encrypted data key with data. Only master key in KMS
  • Key Deletion: 7–30 day waiting period. Cannot cancel after deletion completes
  • Multi-Region Keys: primary + replicas in different regions. Same key ID. Used for global DynamoDB/Aurora
  • KMS throttle: 5,500–30,000 req/s depending on region. Use S3 SSE-KMS carefully at high throughput

  • MGN (Application Migration Service) replaced SMS (Server Migration Service) — use MGN
  • Backup Vault Lock: once locked in Compliance mode, even root cannot delete backups

Classic Architecture Patterns

  • Stateless apps: session state in ElastiCache or DynamoDB. Any instance can handle any request
  • 3-tier web: Route 53 → CloudFront → ALB → EC2/ECS (Auto Scaling) → RDS Multi-AZ + ElastiCache
  • Serverless web: CloudFront → S3 (static) + API GW → Lambda → DynamoDB
  • Event-driven: S3 event → SNS → SQS → Lambda (fan-out + decoupling)
  • Microservices: ECS/EKS + ALB path routing + SQS between services

High Availability Patterns

  • Multi-AZ = same region, different data centres → protects against AZ failure
  • Multi-Region = different regions → protects against region failure
  • Active-Passive: one region active, another in standby (Pilot Light or Warm Standby)
  • Active-Active: both regions serving traffic (Route 53 Weighted/Latency routing)
  • RDS: Multi-AZ for failover, Read Replicas for read scaling

Cost Optimization Patterns

  • Use Savings Plans (flexible) or Reserved Instances (specific instance type) for 1-3 yr commitments
  • Spot Instances for stateless, fault-tolerant batch workloads (up to 90% savings)
  • S3 Intelligent-Tiering: unknown access patterns → auto-moves data to cheapest tier
  • Right-size: use Compute Optimizer or Cost Explorer for recommendations
  • Delete unattached EBS volumes, unused Elastic IPs, idle NAT Gateways
  • CloudFront in front of S3: reduces S3 request costs + data transfer costs

Decoupling Patterns

  • Tight coupling: sync request/response. Fragile. Hard to scale independently
  • Loose coupling: async messaging (SQS/SNS/EventBridge). Resilient. Each service scales independently
  • Fan-out: SNS → multiple SQS queues. One event processed by multiple consumers independently
  • Throttle spikes: SQS absorbs bursts, consumers process at their own pace
